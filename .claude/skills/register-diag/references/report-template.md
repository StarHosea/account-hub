# 诊断报告模板

```markdown
## 诊断摘要
- 邮箱 / 时间 / 失败原因（brief 原文）
- 卡点：`failed_step` + 最后 URL
- 分类：<超时|风控|selector|验证码|代理|状态机|信息不足|其他>（须有字段依据）
- 存证：<有/无> — recording_missing 则 DOM 根因标「无法判断」

## 证据（逐条引用 brief 字段，禁止无出处断言）
- visible_ui：（buttons / hints 原文）
- manifest 末 2～3 步：（stepId + note）
- logs_tail：（关键行原文）
- pageState / reason：（若有）

## 证据审计
- [ ] 能确定卡在哪步？
- [ ] 能描述当时可见 UI？
- [ ] 能说明脚本 selector/操作结果？
- [ ] 状态机/网络/杀进程原因可查？
- 缺口：（任一未勾选 → 本轮只做采集增强，不做修复）

## 根因判断
- **置信度**：<已证实|高|低|无法判断>
- **判断**：（每条格式：「因为 [字段=值] → 所以 …」）
- **待验证假设**：（仅低/无法判断时；每条附验证命令或 CDP 步骤）
- **诊断缺口**：（置信度非已证实时必填）

## CDP 验证（DOM/selector/文案/状态机相关时必填；未做则修复建议留空）
- 目标陈述：
- snapshot 关键行：
- 操作与结果：
- 结论：已证实 / 未证实

## 修复建议
> 规则：仅「已证实」且 CDP 已验证（如适用）才填第 1 节；否则只填第 2、3 节。

1. **代码改动**（仅已证实 + CDP 已验证）
2. **必须先做**（采集增强 / CDP 验证 / 重跑）— 证据不足时只填此项
3. **基础设施**（须有 logs 原文依据）

## 闭环：增强采集
- brief 是否足够：<是 / 否 — 缺什么字段>
- 代码/Skill 改动：（具体文件、record 字段、brief 暴露名）
- 验证：（改完后如何确认新 brief 能答出证据审计项）

## 下一步
- [ ] 补采集（证据不足时唯一优先项）
- [ ] CDP 验证（DOM 相关）
- [ ] 改代码（仅已证实后）
- [ ] run-one 全链路复测
```
