# 注册机代码详解

这份文档面向想读懂这套 OpenAI / ChatGPT 账号注册机的人。它不堆架构图，而是顺着代码把「一个账号是怎么从零被注册出来的」讲清楚：谁在指挥、谁在干活、每一步在做什么、为什么这么做、有哪些规则和坑。

读完你应该能回答：注册机为什么是一半 Python 一半 Node？出口 IP 是怎么保证一号一个的？邮箱和验证码走不走代理？浏览器里到底点了哪些按钮？账号什么时候进号池、什么时候被丢进异常清单？

---

## 一、先看全局：两个进程，一个指挥一个干活

注册机不是一个程序，而是**两套代码协作**：

- **Python 那半是「大脑」**，代码在 `services/register/openai_register.py`。它负责所有的决策和资源：并发调度、给每个账号准备一条干净的代理线路、去邮箱服务商那里领一个邮箱、盯着注册结果、检测账号有没有试用资格、最后把账号写进号池。
- **Node 那半是「手」**，代码在 `node_engine/` 目录。它只做一件事：打开一个真实浏览器，在 ChatGPT 的注册页上一步步点击、填写、收验证码、设密码、开两步验证。

两者是**主从关系**，不是二选一。Python 每要注册一个账号，就用 `subprocess` 现起一个 Node 子进程，把这个账号需要的信息（邮箱、代理、是否开 2FA 等）打包成一段 JSON 丢给它；Node 跑完这一个账号就退出，下一个账号再起一个新的 Node 进程。所以你会看到很多短命的 node 进程来来去去，每个只服务一个账号。

它们之间靠「进程的标准输入输出」通信，用的是一种极简协议：**每一行是一个 JSON**（业界叫 NDJSON）。Node 干活时，每完成一步就往标准输出打一行 JSON 汇报进度；Python 那边逐行读、逐行解析，据此打印日志、更新前端进度、最后收下结果。反过来，当 Node 在浏览器里卡在「请输入邮箱验证码」时，Python 会把从邮箱收到的验证码，经由 Node 进程的标准输入喂回去。

这里有个贯穿全文的关键分工，先记住，后面处处印证：

- **代理线路怎么选、出口 IP 怎么保证不重复，全是 Python 说了算**，Node 只是拿到一个算好的代理地址、照着用。
- **收邮件、取验证码是 Python 直连去做的，永远不走代理**；只有浏览器本身走代理。
- **账号能不能进号池、要不要回填旧密码，全是 Python 的判断**，Node 只负责把注册结果（比如登录令牌）报回来。

## 二、大脑：Python 主控（`openai_register.py`）

### 2.1 一个账号的完整旅程：`worker(index)`

整套流程的主函数是 `worker(index)`，`index` 就是这个账号在本轮里的编号，日志里 `[任务3]` 的 3 就是它。跟着这个函数从头走一遍，你就懂了大半：

1. **登记「开始」状态**。先在进度表里把这个编号标成「运行中／任务启动」，前端能立刻看到有个新任务冒出来。

2. **生成一个身份**。调 `build_identity`，按配置里的地区池（默认 `["US"]`）造出一个虚拟身份——国家、语言、生日、姓名这些。后面填注册表单、决定浏览器语言都靠它。

3. **拿一条能用的代理线路**（`_acquire_working_proxy`，这是重头戏，见 2.2）。返回两样东西：账号专属代理地址，和这条线路的**出口公网 IP**。

4. **把代理转成浏览器能用的格式**（`_browser_proxy_url`）。这一步有个坑：项目里代理默认是 `socks5h://` 带账号密码的，但 Chromium 不支持「带认证的 socks5」，所以这里会强制改写成 `http://` 形式，并把用户名密码做 URL 编码。无法解析的代理就返回空串，让浏览器直连（并打一条黄色警告）。

5. **领一个邮箱**。先看用的是哪种邮箱服务（决定日志里说「获取」还是「生成」），然后 `mail_provider.create_mailbox` 去要一个邮箱回来。这一步失败（比如邮箱服务挂了、或返回的地址是空的），就直接判这个账号失败：统计里 done 和 fail 各加一，打红字日志，清掉进度条目，返回。领到了就把邮箱登记进进度表，日志里报「邮箱获取完成[标签]: xxx@xxx.com」。

6. **交给浏览器干活**（`_run_browser_job`，见 2.3）。这一步会起 Node 子进程、把活干完，返回三样东西：结果数据、错误信息、以及「部分完成状态」。

7. **处理结果**。如果拿到了有效的登录令牌（accessToken），说明注册基本成功了，接下来判断账号「值不值得留」：
   - 调 `trial_check.check_eligibility` 检测这个账号有没有试用资格。
   - **没有试用资格**：账号不进号池，而是被 `register_abnormal_service.add` 丢进「异常清单」，记下原因、令牌、密码等，邮箱标记为已用，统计里算一次失败，日志打黄字「注册成功但无试用资格，已入异常清单，不入号池」。
   - **有资格**（或压根没开检测、或检测本身报错——这几种都按「算合格」处理，见规则一节的 fail-open）：调 `_build_account` 把账号组装好，**连同出口 IP 一起**写进号池。

8. **凭据回填与清理旧条目**。这一步是为「同一个邮箱之前注册过、这次是重登或重设」准备的：
   - 如果这次浏览器没产出新密码／新 2FA 密钥，就用本地已存的旧值补上——**但只在新值为空时才补，绝不用空值覆盖已有的密码**。
   - 然后把这个邮箱名下的旧账号条目删掉（旧的登录令牌早失效了），避免号池里一个邮箱留一堆陈旧记录。注意顺序：**必须先读出旧凭据、回填完，再删旧条目**，否则旧密码就跟着旧条目一起被删了。

### 2.2 重头戏：怎么保证「一号一个 IP」（`_acquire_working_proxy`）

这是整个注册机最讲究的一块，目标是：**本轮里每个账号用的出口公网 IP 都不一样**。为什么重要？因为一批账号如果都从同一个 IP 冒出来，极易被风控识别为批量注册。

先理解底层手段。项目用的是 ipweb 这种「动态住宅代理」，它的代理地址里有一段叫 SID 的会话标识，**换一个新 SID，网关就会给你换一条出口线路、大概率换一个出口 IP**。所以「换 IP」在代码里等于「换 SID 重新解析代理」。这个改写动作由 `_resolve_account_proxy` 完成——它每被调用一次，只要开了 ipweb 轮换，就会换国家段、生成一个**全新的 SID**。

但「换了 SID」不等于「真换到了新 IP」，也不等于「这条线路真的活着」。所以要**探活 + 去重**，逻辑是这样的：

- **没配代理**：直接返回空，账号走直连，出口 IP 也是空。

- **关掉了探活**（配置 `ip_probe_retries` 设成 0）：只解析一次代理直接用，不探活也不去重，出口 IP 记为空。这是「我不在乎一号一 IP」时的省事模式。

- **正常模式（探活开着）**：进一个循环，最多试 `ip_probe_retries` 次（默认 6，只有 ipweb 模式才会重试多次，固定代理只探一次）。每一轮：
  1. `_resolve_account_proxy` 换一个新 SID，拿到一条候选代理。
  2. `_probe_exit_ip` 拿这条代理去 GET `api.ipify.org`，问「我现在的出口 IP 是多少」。请求走代理（这是**唯一一处非浏览器却走代理的地方**，因为只有走代理才能验证这条出口线路真的通），拿到就是这条线路的真实出口 IP，失败返回空。
  3. 探到 IP 后，拿一把全局锁去查这个 IP 有没有被本轮别的账号占用：
     - **没被占用**：登记进「已用 IP 集合」，成功返回。
     - **被占用了（撞号）**：说明这条 SID 换出来的 IP 跟别人重了，打黄字「已被占用（撞号），换 SID 重试」，continue 进下一轮。
  4. 如果这一轮压根没探到 IP（线路没通），打黄字「探活失败，换 SID 重试」，进下一轮。

- **重试都用光还没拿到干净 IP**：打黄字「多次换 SID 仍未探到未占用的新 IP，回退沿用最后一条代理继续（本账号可能与他号共用 IP）」，然后**仍然带着最后那条代理**返回（只是出口 IP 记为空）。也就是说，最坏情况下账号照样能注册，只是可能跟别人共用 IP，不会因为凑不齐独立 IP 就放弃这个账号。

- **固定代理（非 ipweb）撞号**：换 SID 没意义（固定代理本来就共用一个 IP），所以撞号也照用。

**这里有个你要特别知道的日志细节**：只有「换过线之后才成功」「撞号」「探活失败」「回退」这几种情况会把出口 IP 打进日志。**第一次就探活成功的账号（最常见的顺利情况）不会打印出口 IP**——代码里那句判断是「试的次数大于 1 才打印」。所以如果你盯着实时日志想逐个核对每个账号的出口 IP，顺利的账号是空的。但别担心，出口 IP 一定被存进了号池账号记录的 `exit_ip` 字段，可以从落库数据里核对去重效果。

还有一个全局约定：**每开始新一轮注册，都会清空「已用 IP 集合」**（`reset_used_exit_ips`），不然上一轮占用的 IP 会让这一轮误判撞号。

### 2.3 怎么把活交给 Node，又怎么收结果（`_run_browser_job` / `_spawn_worker`）

`_run_browser_job` 是 Python 和 Node 的交界处。

**先组一个 job**：把这个账号浏览器阶段需要的一切打包成一个字典——邮箱、算好的浏览器代理地址、是否开 2FA、是否无头、目标站点、超时毫秒数、浏览器语言（取身份的第一段语言）。如果这个邮箱本地存过密码／2FA（说明以前注册过），就额外塞进 `loginPassword` 和 `existingTotpSecret`，让浏览器遇到「这邮箱已注册」时能先用旧密码登录兜底，并打日志说明注入了哪些凭据。组完还会记一个「基线时间」——注册前邮箱里最新一封邮件的时间，后面判断「哪封才是这次的新验证码」要用。

**再起进程**（`_spawn_worker`）：用 `subprocess.Popen` 起 node，命令行是 `node worker.js <job的JSON>`——注意 **job 是作为命令行参数（argv）传进去的**，不是走标准输入。同时把 License Key 通过环境变量传给子进程，工作目录设成 `node_engine`，保证 Node 内部的相对 require 找得到文件。

**然后泵数据**：逐行读子进程的标准输出，每行 `json.loads` 出一个事件，看它的 `type` 分派：
- `ready`：子进程就绪。
- `step`：调 Python 侧的 `step()`，既更新进度表又打印一行日志。
- `log`：普通日志。
- `result`：收下结果数据（里面有 accessToken）。
- `error`：收下错误信息，可能附带「部分完成状态（partial）」。
- 当浏览器需要验证码时，Python 这边去邮箱把码取回来，再经由**标准输入**把码写回给 Node（`_send_line`）。

**超时与善后**：整个浏览器阶段受 `register_timeout`（默认 300 秒）约束，超时就终止子进程。子进程都登记在一个活跃集合里，方便统一管理和杀进程。返回给 `worker` 的是「结果 / 错误 / 部分状态」三件套，让上层决定这账号算成功还是失败。

### 2.4 邮箱和验证码为什么不走代理

`worker` 里给邮箱模块传配置时，其实把代理也一起传了进去（`_mail_config`）。但这是个**看着矛盾、实则安全**的设计：邮箱模块内部会强制把代理清空（下一节 Node/邮箱部分会详讲）。也就是说，**Python 主控这边领邮箱、轮询收验证码，全程直连，不碰账号的代理**。理由很实际——邮箱服务和账号的出口线路是两码事，收件走代理只会更慢更容易失败，还可能把邮箱服务的访问也暴露到住宅 IP 上。所以整套系统里，**代理只服务于「浏览器出口」和「出口 IP 探活」这两件事**，其余一律直连。

### 2.5 日志和进度是怎么出来的

有两个函数要分清：
- `log(text, color)` 只做两件事：如果挂了 SSE 回调就推给前端一份，然后加锁打印到终端（带时间戳和颜色）。它**不进进度表**。
- `step(index, text, color)` 是「双写」：先更新进度表里这个任务的当前步骤，再调 `log` 打印一行 `[任务N] ...`。前端表格看到的「当前进行到哪一步」就是它更新的。

进度表本身是一个内存字典，加锁保护，前端要展示时通过 `progress_snapshot` 拿一份快照去组装 SSE。

### 2.6 并发模型

`worker(index)` 是单账号入口，外层注册服务会用线程池按编号并发跑很多个。所有跨账号共享的东西都上了锁：进度表、统计、终端打印、已用 IP 集合、活跃子进程集合。其中「已用 IP 集合」是实现全局一号一 IP 去重的关键——它被所有并发账号共享，谁先占了某个出口 IP，后面撞上的就得换 SID。每开新一轮会重置进度表和已用 IP 集合。

### 2.7 主要配置项一览

这些都在 config 默认字典里，实际运行可被覆盖：

- `register_timeout`（300 秒）：单账号总超时，超了杀浏览器判失败。
- `ip_probe_retries`（6）：出口 IP 探活重试次数，0 表示彻底关掉探活（也就没有一号一 IP 和 IP 日志了）。
- `ip_duration`（120 分钟）：出口 IP 的粘性时长，让同一个 IP 在注册全程和后续操作里尽量保持不变。
- `ipweb_rotate`（默认开）：是否启用 ipweb 的一号一 IP 轮换（换国家段 + 换新 SID）。
- `enable_2fa`（默认开）：注册成功后要不要给账号开两步验证。
- `cloakbrowser_license`：浏览器引擎的授权，也可用环境变量传。
- `node_bin`（"node"）：node 可执行文件路径，容器里可能不在默认 PATH。
- `headless`（默认关）：无头浏览器开关，服务器上要开。
- `chatgpt_url`：目标站点。
- `regions`（["US"]）：身份和语言的来源地区池。
- `proxy`：基础代理地址，空就直连。
- `mail`：邮箱服务配置。

## 三、邮箱与代理改写（`mail_provider.py`、`fingerprint.py`）

### 3.1 邮箱模块为什么「明明传了代理却不走代理」

前面提过，主控给邮箱模块传配置时把 `proxy` 也带了进去，但邮箱模块内部有一个 `_config` 函数，它在整理配置时**硬编码把 `proxy` 覆盖成空串**，并在注释里写明「收件取件地址永不走代理（注册流程的 socks5 代理与收邮件解耦）」。这就是整套系统「收邮件一律直连」这条铁律的落点——不管上层怎么配，到了真正发起邮箱请求那一刻，代理已经被清成空的了。

所以这里没有矛盾：主控图省事统一传了同一份配置，邮箱模块负责把不该继承的代理摘掉。想改这条规则，就得动 `_config` 这一行，而不是去改上层。

### 3.2 邮箱的领取、收码与生命周期

邮箱模块支持不止一种来源，常见的是「API 邮箱池」和「CloudMail 按需生成」两类，用类型常量区分。主控里那句「获取还是生成」的措辞，就是根据当前启用的是哪种 provider 决定的——池子里捞现成的叫「获取」，CloudMail 现造一个叫「生成」。

`create_mailbox` 负责要一个邮箱回来，返回的结构里带着地址、标签、取件地址这些字段，主控靠地址判断领取是否成功、靠标签在日志里标注。收验证码是一个「轮询」过程：按配置的间隔和超时反复去取件地址拉邮件，配合前面记的「基线时间」筛掉注册之前就躺在收件箱里的旧邮件，只认注册之后到达的新验证码。注册结束后，邮箱还有标记结果、释放这些收尾动作，让邮箱服务知道这个邮箱用过了、成没成。

### 3.3 一号一 IP 的底层改写：`rotate_ipweb_proxy`

这是「换 IP」真正发生的地方。ipweb 的代理地址是有固定结构的，用户名那段形如 `B_<账号id>_<国家>_<州>_<城市>_<时长>_<SID>`（真实样例长这样：`B_88059_IN_1106_10126_10_x2HnZNuU`，冒号后面还跟着时长）。`rotate_ipweb_proxy` 做的就是在这个结构上动手脚：

- 只有当代理主机名是以 ipweb 的域名结尾、且用户名符合上面那个格式时才改写，否则原样返回（保证对非 ipweb 代理无副作用）。
- 把**国家段换成这个身份对应的国家**（比如身份是美国就换成对应国家码），实现「不同账号从不同国家段出去」。
- 把**最后那段 SID 整个换成一个全新的随机串**（由一个随机字母数字生成器产出，长度跟原 SID 一致），这是让网关换一条出口线路、从而换出口 IP 的关键。
- 顺便**清空州、城市段**，并允许用参数**覆盖时长**（对应配置里的 IP 粘性分钟数）。清空州市是为了不把出口限死在某个小地理范围，给网关更大的选路空间。

理解了它，就理解了 2.2 里「每解析一次代理就换一个新 SID」到底换了什么：换的就是这段 SID，网关据此给一条新线路，再由探活去验证这条线路的真实出口 IP。

## 四、手：Node 浏览器引擎（入口与基础设施）

### 4.1 子进程入口 `worker.js`

Node 这半的入口是 `worker.js`。它被 Python 用 `node worker.js <job的JSON>` 起起来，所以**它做的第一件事就是把命令行第三个参数解析成 job 对象**（邮箱、代理、是否 2FA、无头、目标站点、超时、语言，以及可能有的登录密码和已存 2FA 密钥）。

拿到 job 后它按顺序做三件事：

1. **起浏览器**：调 `launchSession(job.proxyUrl, ...)` 打开一个带指纹、按需走代理的浏览器会话。
2. **挂上「接收主控指令」的耳朵**：用 readline 监听自己的标准输入，主控随时可能下发一行 JSON（最典型的就是邮箱验证码），交给一个处理函数按需消费。这就是 2.3 里「验证码经标准输入喂回来」的接收端。
3. **开跑注册**：调 `runRegister(session, job, {emitStep, emitResult, emitError, emitLog})`，把注册主流程交出去，同时把四个上报函数注入进去，让流程内部每一步都能往主控汇报。

跑完之后按结果分三种情况上报：拿到登录令牌就 `emitResult` 报成功；只完成了一部分（比如注册成功但 2FA 没设完）就 `emitError` 并带上「部分状态」；彻底失败就 `emitError` 报错误原因。无论哪种，最后都在 finally 里把浏览器会话关掉、进程退出——**一个 Node 进程只服务一个账号，干完就干净退出**。

### 4.2 通信协议 `protocol.js`

这个文件很小，就是把「往标准输出打一行 JSON」这件事封装成几个语义明确的函数：`emitReady`（就绪）、`emitStep`（进度步骤）、`emitResult`（最终结果）、`emitError`（错误）、`emitLog`（普通日志）。每个都是拼一个带 `type` 字段的对象、`JSON.stringify` 后加换行写出去。主控那边就是靠这个 `type` 分派的。方向是单向的——**这个文件只管往外报，不管往里读**；往里读（接收主控指令）是 `worker.js` 用 readline 单独做的。

### 4.3 起浏览器与兜底 `cloakbrowser.js`

`launchSession` 是起浏览器的核心。它优先用项目的 CloakBrowser 引擎（一个做了指纹伪装、反检测加固的 Chromium 封装），把几样东西配上去：固定的指纹种子（保证同一账号指纹稳定）、geoip（按出口 IP 自动匹配时区和语言，这也是为什么代理出口 IP 要准）、无头开关、以及代理。代理如果有值就整条带认证的地址传进去。

关键是它有**兜底**：万一 CloakBrowser 引擎启动失败或加载不出来，就退到普通的 playwright-core Chromium 继续跑（`launchFallbackChromium`）。这条回退路径要自己把代理地址解析成 playwright 认的格式（`toPlaywrightProxy` 负责把 `http://user:pass@host:port` 拆成 server／username／password 三段）。这样即便主引擎出问题，账号也还有机会用备用内核注册成功，日志里会提示「已切换备用模式」。

License 是通过环境变量传进来的（主控 spawn 时注入），CloakBrowser 引擎启动时读取。

### 4.4 工具 `utils.js`

随机资料、密码生成、TOTP（两步验证的动态码）计算都在这里。文件头注释特意写了「移植自另一个项目，**去掉了 IpWeb 会话串生成**」——这句话很重要，它印证了：**Node 侧曾经有过 ipweb 相关逻辑，现在已经彻底删掉了**，代理决策完全归 Python。所以你在 Node 侧搜到的任何 ipweb／出口 IP 字样都只是注释，没有活代码。

## 五、浏览器里到底做了什么（`register.js`）

`register.js` 是整个引擎最大的一个文件（约 1760 行），承载了浏览器里所有的实际操作。它内部可以理清成**四条线**：一堆拟人化的底层工具、全新注册主线、邮箱已注册时的登录兜底线、以及最后设密码开 2FA 的 step8。下面按线讲，函数后面括号里是它在文件里的大致行号，方便你对照源码。

### 5.1 底层工具：让操作「像人」而不是「像脚本」

风控很在意操作是不是机器人式的，所以这里有一层拟人化封装：`humanClickByText`（按文字找按钮并模拟人手点击，行 28）、`humanType`（一个字符一个字符地敲、带节奏，行 104）、`clickButtonRobust`（多种方式尝试点一个按钮直到点中，行 386）、`clickByTestId`（按测试 id 定位点击，行 93）。还有一批用于「等待和判断」的：`openWithRetry`（打开页面，住宅代理慢或抖动时重试，行 212）、`firstVisible`（一组候选里挑第一个可见的，行 69）、`pageText`（取页面文本用于判断，行 115）、`waitForAuthReady`（等鉴权就绪，行 374）、`snapshot`（存一份现场快照，行 196）。这些是后面所有流程的积木。

### 5.2 全新注册主线：`registerChatGPT`（行 876）

这是一个全新邮箱从打开注册页到拿到令牌的主编排。串起来的步骤大致是：`openWithRetry` 打开注册页 → `fillProfile`（行 1199）填基本资料 → `fillBirthday`（行 262）填生日 → `waitForCodePage`（行 155）等验证码页出现，`isOnCodePage`（行 162）确认确实到了这一步 → `fillCode`（行 1076）把验证码填进去、`submitCodeForm`（行 1127）提交。

验证码提交后有个关键的**分支判断** `detectPostCodeBranch`（行 1171）配合 `hasLeftCodePage`（行 1151）：提交完到底进了哪个页面？可能直接注册成功，可能还要补资料，可能弹出别的要求。流程据此决定下一步走向。再往后是 `checkConsentIfAny`（行 1222）处理可能出现的条款同意、`waitForSuccess`（行 1245）按成功标志确认注册完成、`readAccessToken`（行 238）从会话里把登录令牌抠出来。中途可能有欢迎弹窗挡路，`dismissWelcomeOverlays`（行 1339）负责把它们关掉。

验证码是从哪来的？就是 4.1 说的——浏览器走到验证码页时，主控那边已经在轮询邮箱，收到后经标准输入把码送进来，`fillCode` 拿到就填。

### 5.3 邮箱已注册的兜底线（当前开发分支的主战场）

如果这个邮箱其实之前注册过，打开后会落到登录页而不是注册页，这时走另一条线，入口是 `loginChatGPT`（行 538）。它会用主控注入的「本地已存密码」尝试登录，然后根据遇到的情况分头处理：

- **触发邮箱二次验证**：`isMfaEmailChallenge`（行 670）先识别出「这是一道邮箱 MFA 质询」，然后 `tryLoginMfaEmailFallback`（行 686）走邮箱验证码兜底——同样是等主控经标准输入把邮箱里的码送进来再填。这条 `login-mfa-email-fallback` 正是当前分支 `codex-openai-mfa-email-fallback` 重点在做的事，配套还有同名的单元测试。
- **触发 TOTP 两步验证**：`handleLoginTotpPrompt`（行 742）用主控注入的已存 2FA 密钥算出动态码填进去。
- **密码不对**：`forgotPasswordFlow`（行 421）走「忘记密码」重设流程——触发重设、收邮件里的重设码、设新密码。
- 登录并处理完之后，`secureExistingChatGPT`（行 820）负责已有账号的安全设置收尾。

### 5.4 设密码 + 开两步验证：step8（行 1529）

注册／登录成功之后的最后一大步，是给账号补齐安全设置，主函数是 `step8_setupPasswordAnd2FA`（行 1529）。它要先进到账号的安全设置页：`openSettings`（行 1362）打开设置 → `openSecurityTab`（行 1423）切到安全标签 → `waitForSecurityTabReady`（行 1386）等这个标签真正加载好。

然后做两件事。密码方面，`hasExistingPassword`（行 1506）先判断这个账号是不是已经有密码了，据此决定是设新密码还是跳过。两步验证方面，如果账号上已经有旧的 2FA，可能需要 `disable2fa`（行 1462）先关掉再重开；开启 2FA 的过程中会拿到一个 TOTP 密钥，`extractTotpSecret`（行 1739）负责把这个密钥抠出来——**这个密钥必须被抓到并回传落库**，否则以后就没法给这个账号算动态码登录了。整个 step8 是否执行、要不要开 2FA，受主控 job 里的 `enable2fa` 控制。

step8 这块对容错要求很高（设置页 UI 经常变、加载慢），所以配了独立的单元测试（`step8-2fa-tolerance`）专门测它的容忍度。诊断辅助上还有 `dumpUi`（行 1271）在出问题时把当前 UI 存下来、`clickMarked`（行 1289）点击标记过的元素。

## 六、状态探测与存证（`auth-state.js`、`dom-recorder.js`、`selectors.js`）

上面那些流程之所以能在「注册页 / 登录页 / 验证码页 / 2FA 页 / 成功页」之间不迷路，靠的是**状态探测**这一层。`auth-state.js` 把「我现在到底在哪一步」这个判断统一收口——不再让每个流程各自靠零散信号猜，而是集中根据页面的 URL、DOM 结构、关键文案综合判断当前处于哪个状态，再由 `register.js` 里的流程据此决定下一步。这样页面一旦变化，只要改这一处判断逻辑即可。

`dom-recorder.js` 是「黑匣子」。它由环境变量 `REG_RECORD_DIR` 控制开关：开启后，注册的每一步都会把当时的 DOM 录一份下来，方便事后回放排查。它的策略是**成功即删、失败留证**——顺利跑完的账号不留垃圾，出问题的账号把现场保留下来供你复盘。这在注册这种「线上偶发失败、难复现」的场景里特别有用。

`selectors.js` 则是把所有页面元素的定位选择器集中定义在一处，按页面／步骤组织。UI 上的按钮、输入框改了，来这里改选择器就行，不用翻遍流程代码。

## 七、注册规则总清单

把散落在各处的业务规则集中列一遍，方便查：

1. **一号一 IP**：开启 ipweb 轮换时，每个账号换国家段 + 全新 SID，探活拿到出口公网 IP 后做全局去重；撞到别的账号已占用的 IP 就换 SID 重试，只有拿到全新 IP 才放行。
2. **探活重试与回退**：默认最多试 6 次；全失败也不放弃账号，回退沿用最后一条代理继续（此时可能与他号共用 IP，日志会提示）。把重试次数设为 0 就整个关掉探活，代价是没有一号一 IP、也没有出口 IP 日志。
3. **出口 IP 粘性**：默认 120 分钟，让同一个 IP 覆盖注册全程和后续单次操作。
4. **邮件永不走代理**：收邮箱、取验证码全程直连，由邮箱模块内部强制清空代理保证。代理只服务浏览器出口和出口 IP 探活。
5. **邮箱已注册的兜底优先级**：先用本地存的密码登录 → 遇邮箱 MFA 走邮箱验证码兜底 → 遇 TOTP 用已存密钥 → 密码错走忘记密码重设。
6. **两步验证**：默认给注册成功的账号开 TOTP，并把密钥抓下来落库；已有旧 2FA 时先关再开。
7. **试用资格是入号池的门槛**：没资格的账号进异常清单、不入号池、不自动激活；有资格、或没开检测、或检测本身报错（fail-open 按合格）才入号池。
8. **凭据回填不覆盖**：这次没产出新密码／新 2FA 就用本地旧值补，但绝不用空值覆盖已有凭据；回填之后再删同邮箱的旧条目。
9. **单账号超时**：默认 300 秒，超时判失败并杀掉浏览器子进程。
10. **出口 IP 日志有缺口**：第一次探活就成功的账号不打印 IP，但 IP 一定落库到账号的 `exit_ip` 字段，核对去重看落库数据最可靠。

## 八、一次成功注册，从头到尾发生了什么

把两个进程合起来，顺一遍最顺利的一条路：

1. 主控的 `worker(3)` 启动，先造一个美国身份。
2. 进入取代理环节：换一个新 SID 解析出 ipweb 代理，拿它去 ipify 探活，问出真实出口 IP，确认这个 IP 本轮没人用过，登记占用。（这一步顺利时不打 IP 日志，但 IP 记在手里了。）
3. 主控直连去邮箱服务领一个邮箱，日志报「邮箱获取完成」。
4. 主控把邮箱、算好的浏览器代理、是否开 2FA 等打包成 job，`node worker.js <job>` 起一个 Node 子进程。
5. Node 解析 job，用 CloakBrowser 带着代理和指纹打开浏览器，挂上标准输入监听，开始跑注册主线：打开注册页、填资料、填生日、走到验证码页。
6. Node 每完成一步就往标准输出打一行 JSON，主控逐行读、逐行打日志、更新前端进度。
7. 走到验证码页，主控这边已经在轮询邮箱，收到验证码后经 Node 的标准输入把码送进去，Node 填码、提交、判断分支、确认注册成功。
8. Node 进入 step8：进安全设置、设密码、开两步验证、抓下 TOTP 密钥，最后从会话里读出登录令牌，`emitResult` 把令牌、密码、密钥一起报给主控，然后关浏览器、退出进程。
9. 主控收到结果，用令牌检测试用资格，合格。
10. 主控把账号连同出口 IP 组装好写进号池，需要的话用本地旧值回填空缺凭据、删掉这个邮箱的旧条目，统计加一，这个账号就注册完成了。

*本文基于当前代码梳理（分支 `codex-openai-mfa-email-fallback`）。核心结论：ipweb 出口 IP 的探活、换 SID、一号一 IP 去重全部只在 Python 侧，Node 侧对代理是纯消费、无重复实现；收邮件永不走代理，代理只服务浏览器出口和出口 IP 探活。`register.js` 里具体每一步的 DOM 操作细节以源码为准，本文讲到每个函数的职责与所在行号，便于对照阅读。*
